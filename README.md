## 3. Обязательные требования

- [ ] Обнаружение цели за ≤ **100 мс** с момента появления в кадре
- [ ] Присвоение **уникального ID** каждой обнаруженной цели
- [ ] Передача координат (X, Y, угол, дистанция) в управляющий контроллер с частотой **≥ 10 Гц**
- [ ] Сохранение трека при пропадании цели до **3 секунд** (фильтр Калмана)
- [ ] Автоматический возврат в режим **«ПОИСК»**, если цель не появилась за **5 секунд**
- [ ] Фильтрация **статических объектов** (деревья, столбы, строения)
- [ ] Работа при **дожде, сумерках, засветке** (без переобучения, только аугментация)
- [ ] **Полная автономность** — вся логика на борту, без внешнего сервера
- [ ] Логирование **сырых кадров** при ошибках (режим «чёрного ящика»)
- [ ] Поддержка **MIPI‑CSI / USB‑камер** с частотой ≥ 30 FPS
- [ ] Обновление модели через **USB‑накопитель или OTA (Wi‑Fi)**

## 4. Сценарий работы (как пользователь видит систему)

### Старт
- На экране HUD: **сетка прицела**, компас, индикатор **«ПОИСК»**
- Алгоритм сканирует кадр, отображает рамки вокруг всех движущихся объектов

### Наведение на цель
- Пользователь кликает на цель (или цель выбирается автоматически по приоритету)
- Система фиксирует её признаки **(ReID‑вектор)**
- Переключение в режим **«ТРЕКИНГ»**
- На HUD появляется **прицельный крест**, который плавно следует за целью
- Серводвигатели отрабатывают с задержкой **< 200 мс**

### Потеря цели
- Цель уходит за препятствие / выходит из кадра
- Крест становится **оранжевым** (статус **«ТРЕК ПОТЕРЯН»**)
- Запускается таймер **5 секунд**
- Если цель не появляется → крест гаснет, система переходит в режим **«ПОИСК»**

### Возврат цели
- Цель появляется снова
- Система сравнивает текущие объекты с сохранённым ReID‑вектором
- При совпадении → **мгновенный захват** (за 1 кадр)
- Крест становится **зелёным**, наведение продолжается без участия пользователя

### Статусы HUD
| Статус | Цвет | Описание |
|--------|------|----------|
| **СКАНИРОВАНИЕ** | Синий | Активный поиск цели в кадре |
| **ЗАХВАТ** | Зелёный | Цель зафиксирована, идёт наведение |
| **ТРЕКИНГ** | Жёлтый | Устойчивое сопровождение цели |
| **ПОТЕРЯ** | Оранжевый | Цель пропала, идёт таймер ожидания |
| **ПОИСК** | Красный | Активный поиск по всей сцене |

## 5. Процесс обработки кадра: YOLO + Трекер

### 5.1. Роли компонентов

| Компонент | Что делает | Когда запускается |
|-----------|------------|-------------------|
| **YOLO (детектор)** | Находит все объекты на кадре, выдаёт координаты боксов и классы | Первым, каждый кадр |
| **Трекер** | Связывает детекции на разных кадрах, присваивает уникальные ID, прогнозирует движение | Вторым, каждый кадр (сразу после YOLO) |
| **Постобработка** | Выбирает основную цель, рассчитывает доп. параметры | Третьим, каждый кадр |
| **Управление** | Передаёт команды на сервоприводы / моторы | Четвёртым, каждый кадр |
| **HUD** | Отрисовывает интерфейс для оператора | Пятым, каждый кадр |

### 5.2. Пример кода (упрощённый)

from ultralytics import YOLO
from tracker import BoTSORT

# 1. Инициализация
model = YOLO("yolov8n.pt")
tracker = BoTSORT()
cap = cv2.VideoCapture(0)

while True:
    ret, frame = cap.read()
    if not ret:
        break

    # ШАГ 1: YOLO — детекция
    results = model(frame)
    detections = results[0].boxes.xyxy.cpu().numpy()
    confidences = results[0].boxes.conf.cpu().numpy()

    valid_detections = []
    for box, conf in zip(detections, confidences):
        if conf > 0.5:
            valid_detections.append(box)

    # ШАГ 2: Трекер — ассоциация

5.3. YOLO vs Трекер: разница
Параметр	YOLO (детектор)	Трекер
Что делает	Находит объекты на кадре	Связывает объекты между кадрами
Выдаёт	Боксы без ID: [x1,y1,x2,y2,conf]	Треки с ID: [ID, x, y, w, h, conf]

Прогнозирует движение	Нет	Да
Без него	Рамки "прыгают" от кадра к кадру	Не с чем работать

## Пояснение к столбцам

### Готовые библиотеки / SDK
- **RKNN-Toolkit2** — официальный инструмент Rockchip для конвертации моделей (TensorFlow, PyTorch, ONNX, Caffe, Darknet) в RKNN-формат. Поддерживает квантизацию и оптимизацию для NPU [citation:9].
- **CAM-1126BQ38** — поставляется с готовыми решениями для запуска 2B-моделей (Qwen, Gemma2-2B, Phi2) и глубокой оптимизацией Transformer [citation:9].
- **Banana Pi BPI-SM10** — поставляется с AI-runtime и набором инструментов для переноса проектов. Заявляется аппаратная совместимость с Jetson, что упрощает миграцию кода [citation:6][citation:10].

### Особенности новых платформ

**CAM-1126BQ38** [citation:1][citation:9]:
- Ультракомпактный модуль 38×38 мм (Firefly)
- AI-ISP и AOV 3.0 — обработка изображения на выделенном железе, энергопотребление в режиме ожидания до 1 мВт
- Поддержка 12MP камер и 4K-видео
- Отличное соотношение цена/производительность для дронов

**Banana Pi BPI-SM10** [citation:2][citation:4][citation:6]:
- Первый RISC-V чип с сертификацией RVA23
- **60 TOPS** — самый мощный NPU в списке (может запускать 30B-параметровые LLM с >10 токенов/с)
- 1024-битный векторный блок для ускорения AI-вычислений
- Аппаратная совместимость с Jetson (упрощает перенос проектов)
- Потребление 18–35 Вт — требует серьёзного охлаждения, для дрона не подойдёт, но для стационарной турели — мощный вариант

# Датасеты для обучения системы перехвата и турели

## Общее описание

Для обучения моделей детекции (YOLOv8/v11) в задачах перехвата дронов и наведения турелей требуются специализированные датасеты. Ниже приведён список открытых наборов данных, сгруппированных по типу задачи.

---

## 1. Специализированные датасеты по дронам

### VisioDECT
**Самый свежий и релевантный вариант.**

- **Объём:** 20 924 аннотированных изображения
- **Классы:** 6 моделей дронов (Anafi-Extended, DJI FPV, DJI Phantom, EFT-E410S, Mavic2-Air, Mavic2-Enterprise)
- **Условия съёмки:** Солнечно, облачно, вечер
- **Высота:** 30–100 м
- **Формат разметки:** YOLO, XML, CSV
- **Где взять:** [IEEE Dataport (DOI: 10.21227/n27q-7e06)](https://ieee-dataport.org/documents/visiodect-dataset-aerial-dataset-scenario-based-multi-drone-detection-and-identification) [citation:2][citation:1]
- **Применение:** Обнаружение и идентификация БПЛА в реальных условиях

---

### BirDrone
**Специализированный датасет для задачи "дрон vs птица".**

- **Объём:** 2970 изображений (2617 дронов, 353 птицы)
- **Разрешение:** 640×640 пикселей
- **Аннотации:** 6162 bounding boxes (от 7×14 до 65×182 пикселей)
- **Классы:** Drone, Bird
- **Формат:** YOLOv8-совместимый
- **Где взять:** [IEEE Dataport](https://test.ieee-dataport.org/comment/5881#comment-5881) [citation:3][citation:4]
- **Применение:** Снижение ложных срабатываний турели на птиц

---

### Toy-3
**Датасет на основе игрушечных моделей военной техники.**

- **Объём:** 1593 изображения
- **Аннотации:** 10 374 размеченных цели
- **Классы:** ArmoredCar, FixedWing, Tank
- **Сценарии:** Асфальт, гравий, трава, кусты, деревянные настилы
- **Разрешение:** 1920×1080 пикселей
- **Где взять:** Доступен в рамках исследования [Nature Scientific Reports](https://link.springer.com/article/10.1038/s41598-025-26601-0) [citation:5][citation:6][citation:7]
- **Применение:** Быстрый старт при отсутствии реальных данных

> **Почему работает:** Модель учится на форме и текстуре, а не на масштабе. Дообучение на реальных данных доводит точность [citation:5].

---

## 2. Готовые проекты на GitHub (веса + код)

### Drone-Based Reconnaissance of Military Assets
**Готовый пайплайн для обнаружения военной техники с дрона.**

- **Классы:** Танки, броневики, грузовики, военные и гражданские объекты
- **Модели:** YOLOv8 / YOLOv11 (все варианты: n/s/m)
- **Формат:** Датасет + обученные веса + скрипты
- **Особенность:** Оптимизация для Edge-устройств (Jetson/Coral)
- **Где:** [GitHub - InvictusRex/Drone-Based-Reconnaissance-of-Military-Assets](https://github.com/InvictusRex/Drone-Based-Reconnaissance-of-Military-Assets) [citation:9]
- **Применение:** Можно взять готовую модель и дообучить под свои цели

---

### MOT_Trackingonjetson
**Автоматическая разметка видео через VLM-модель LocateAnything-3B.**

- **Принцип:** Вы пишете текстовый промпт (`"military tank"`, `"armored vehicle"`) — модель сама размечает кадры в YOLO-формат
- **Преимущество:** Не требует ручной разметки
- **Требования:** ~4.5 ГБ VRAM (модель в 8-bit квантизации)
- **Где:** [GitHub - Abhi-Jha345/MOT_Trackingonjetson](https://github.com/Abhi-Jha345/MOT_Trackingonjetson) [citation:13]
- **Применение:** Быстрое создание собственного датасета под произвольные классы

## 5. Примечания по использованию

1. **Для обучения на Mac M1 / Google Colab** датасеты загружаются в стандартных форматах (YOLO, COCO) и не требуют специального железа.
2. **Для переноса на Orange Pi 5 / RK3588S** датасет используется только на этапе обучения. На этапе инференса модель работает с реальным видеопотоком с камеры.
3. **Дообучение на реальных данных** — обязательный этап, так как синтетические или игрушечные датасеты могут давать ошибки в реальных условиях.
